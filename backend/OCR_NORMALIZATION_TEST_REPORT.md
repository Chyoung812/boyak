# OCR/STT 약명 오타 정규화 테스트 리포트

## 기준

- API/함수: `normalize_medicine_names()`
- 대상: DUR `노인주의`/`연령금기` 제품 1,200개에서 생성한 오타 케이스
- 입력 생성: 약봉투/OCR에서 자주 생기는 형태를 가정
  - `정 → 전`
  - `캡슐 → 캡술`
  - `시럽 → 시릅`
  - `에/애`, `레/래`, `펜/팬`, `센/샌`, `텍/택`, `밀리그램/미리그램`
  - 마지막 글자 누락
  - 앞쪽 한 글자 누락
- 안전 원칙: 오타/부분명은 자동 확정하지 않고 대부분 `needs_confirmation`으로 반환

## 대량 테스트 결과

```txt
대상 제품: 1,200개
생성 테스트 입력: 2,556개

matched: 1개
needs_confirmation: 2,555개
not_found: 0개

제품코드 후보 적중: 2,509 / 2,556 = 98.16%
성분코드 후보 적중: 2,554 / 2,556 = 99.92%
Top-1 제품코드 적중: 2,314 / 2,556 = 90.53%
Top-1 성분코드 적중: 2,432 / 2,556 = 95.15%
기대 제품코드가 후보 안에 없는 케이스: 47 / 2,556 = 1.84%
```

## 해석

- 제품코드 기준 98.16%는 후보 목록 안에 원래 제품이 들어간 비율입니다.
- 성분코드 기준 99.92%는 후보 목록 안에 같은 성분 약이 들어간 비율입니다.
- Top-1만 자동 선택하면 제품코드 기준 90.53%라 부족합니다.
- 따라서 실서비스 흐름은 자동 확정이 아니라 후보 확인 UI가 필수입니다.

## 오탐/불일치 원인

대부분 아래 유형입니다.

```txt
1. 같은 성분/함량인데 제조사·브랜드만 다른 제품이 매우 많음
   예: 두타스테리드 0.5mg 제품군, 덱시부프로펜 연질캡슐 제품군

2. OCR/부분 입력이 브랜드 구분 글자를 날림
   예: 두타론정/두타린정/두타모정 → "두타정0.5밀리그램"처럼 축약되면 구분 불가

3. 제품명은 다르지만 성분코드는 같은 후보가 top으로 올라옴
   안전성 검사는 성분코드 중심이면 대체로 같은 위험군이지만, 제품 확정에는 사용자 확인 필요
```

## 적용한 해결책

```txt
1. 오타/부분명은 matched가 아니라 needs_confirmation으로 반환
2. 너무 느슨한 2글자 fuzzy 매칭 제거
3. 후보 목록에는 product_code/ingredient_code를 같이 반환
4. 프론트에서 약봉투와 맞는 후보를 선택한 뒤 /api/safety/check-selected 호출
5. 근거 없는 동일성분중복 경고는 제거
```

## 남은 UI 요구

```txt
status = matched
→ 그대로 진행 가능하지만 후보 확인은 보여줘도 됨

status = needs_confirmation
→ "이 약이 맞나요?" 후보 선택 필수

status = not_found
→ 약봉투 전체명 재촬영/직접입력 요청
```
